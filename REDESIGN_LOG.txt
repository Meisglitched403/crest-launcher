REDESIGN STRATEGY
══════════════════════════════════════════════════════════════════════════════

[GOAL]
  Deliver a professional glassmorphic/material-design UI that can compete
  with Lunar Client and the official Modrinth app in visual polish,
  component quality, and data-density. Dual dark+light themes. Proper file
  architecture. Full backend contract preserved.

[BACKEND PROPOSAL — already discussed, not implemented in renderer yet]
  (Backend is treated as stable during this redesign. These are future
   endpoints to integrate when the backend team enables them.)

  GET /api/instances/playtime?name=        → track time per modpack
  GET /api/instances/last-played?name=     → last session timestamp
  GET /api/news                             → RSS/cached feed
  GET /api/stats/summary                    → total playtime, sessions, mod
                                              count, server count
  GET /api/system/info                      → live CPU/RAM/Net telemetry

  Design note: new UI gracefully falls back to static/mock data when these
               endpoints return 404, so the UI can ship before backend changes.

KEY DECISIONS MADE AND WHY
══════════════════════════════════════════════════════════════════════════════

1. Branch-based: all work happens on ui/redesign-glass-material. master
   remains clean at pre-redesign-backup tag. Both tracked + pushed.

2. Backend contract stable: only the UI renderer is touched. The Python
   REST server on 8765 is untouched (except reading .py files). The new
   views gracefully fall back to static/mock data when endpoints 404.

3. Dual theme design: :root = light (Frost), .dark overrides = Nocturne.
   This is the inverse of the original (dark default) so that
   ThemeProvider/hydration renders the right one.

4. Material-3 elevation system: 5 levels of glass (elev-1 through elev-5)
   with progressive tonal overlay + depth shadow. Matches Lunar/Modrinth
   surface language.

5. Grain overlay: .grain class on fixed background layer; SVG noise data
   URI; opacity controlled by --grain-opacity CSS var (0.04 in dark mode).

6. Page transition: decided to keep the fade/slide (initial y:12 → y:0,
   exit y:-8) at 0.25s, but theme motion level now divides it. Level 0 =
   off, 1 = 0.25s, 2 = 0.2s, 3 = 0.18s.

7. Shell-centered: HomeView is the hero; all other views are scrollable
   pages routed via NavId state. Topbar/Sidebar are always visible.

8. Component codification: framer-motion's motion + AnimatePresence are used
   pervasively. The motion prop on theme controls MotionConfig's reducedMotion.

9. No code comments added unless requested (style rule applied throughout).

10. PrimaryButton and GhostButton now accept optional `icon` prop (child
    rendered before label text). This resolves prop-type mismatch.
